01 — Context

The Problem We're Solving

As our microservice estate grows, three compounding problems are creating measurable engineering drag — affecting delivery velocity, team onboarding time, and operational risk.

📄

Documentation Debt

Architecture documents, onboarding guides, and API references are either non-existent or months out of date. New developers spend 2–4 weeks just reading code before contributing meaningfully.

🔍

No Systematic Code Review

Security reviews, DynamoDB schema audits, and messaging pattern checks (MQ/Solace) are done ad-hoc or not at all. Issues surface in production instead of in development.

🔀

Scattered Knowledge

Understanding of how microservices interact — which service owns which DynamoDB table, which Solace topic triggers what workflow — lives only in the heads of senior engineers.

⏱️

Manual Effort Cost

A senior architect spending 2 days auditing a single service costs the organization significantly. With 20+ microservices, systematic auditing is economically infeasible manually.

02 — Objective

What This POC Answers

This is not a feature build. This is a controlled experiment with a single binary question at the center.

Central POC Question

Can an AI system — specifically Anthropic's Claude — be fed the structural representation of our Java Spring Boot microservices (deployed on AWS EKS, using DynamoDB, Amazon MQ, and Solace messaging) and automatically produce:

"Accurate architecture documentation, security findings, and developer onboarding guides — without any manual engineering effort, and without sensitive source code or credentials ever leaving our internal environment?"
What we are NOT doing: We are not replacing GitHub Copilot, Amazon Kiro, or any existing developer tooling. This POC specifically targets the gap none of those tools address — repository-wide intelligence and automated documentation generation at the architectural level.

The Sanitization Principle — How Code Is Protected

Before any data is sent to Claude, every file passes through a local sanitization layer. The transformation strips all method bodies, credentials, and business logic — retaining only the structural metadata that allows AI analysis.

  private static final String
    DB_PASS = "Pr0d@Secret2024!";

  public Order processOrder(String id) {
    // 340 lines of proprietary
    // business logic...
    // catches Exception silently
  }
}

Security guarantee: The sanitization layer runs entirely on-premises. Claude receives method signatures, class relationships, dependency names, and structural patterns — never passwords, business logic, customer data, or internal infrastructure details.

05 — Security

Data Security Model

Given the sensitivity of enterprise codebases, the security model is non-negotiable and built into the pipeline architecture — not bolted on.

✅ Guaranteed

No Raw Source Code Leaves Network

The sanitizer runs 100% locally. Only structural skeletons (signatures, class names, dependency lists) are transmitted to the Claude API.

✅ Guaranteed

No Credentials Transmitted

The secret scanner runs before any transmission. Detected credentials are replaced with typed placeholders. Transmission is blocked if validation fails.

✅ Guaranteed

Anthropic API Data Policy

Anthropic's commercial API retains logs for 7 days maximum and never uses API data for model training. Enterprise contract adds Zero Data Retention (ZDR) option.

✅ Guaranteed

IAM Least Privilege

The pipeline runs with a dedicated IAM role scoped to exactly the permissions needed: DynamoDB read, S3 write to audit bucket, CloudWatch logs.

What Claude Sees vs. What It Never Sees

Category ✅ Sent to Claude (Safe) ❌ Never Sent (Blocked)
Code Structure Class names, method signatures, return types, annotations Method bodies, business logic, algorithms
Dependencies Library names and versions (from pom.xml) Internal artifact repository URLs, license keys
Infrastructure AWS service names (DynamoDB, SQS), table name structure Account IDs, ARNs, endpoint URLs, VPC details
Messaging Topic/queue name patterns, message schema shapes Broker credentials, connection strings, certificates
Credentials [REDACTED_AWS_KEY], [REDACTED_DB_PASSWORD] placeholders Actual keys, passwords, tokens, secrets of any kind
Business Data Field names, data types, schema shapes Actual values, customer records, PII of any kind
06 — Execution Plan

POC Phases & Timeline

Four sequential phases over 4–5 weeks. Each phase ends with a concrete, demonstrable milestone — not theoretical progress.

PHASE 1 Week 1–2

Target Microservice + Local AWS Environment

Build a realistic Java Spring Boot microservice that mirrors our production characteristics: DynamoDB persistence, Amazon MQ + Solace messaging integration, AWS SDK v3 usage, Docker containerization, and Kubernetes manifests for EKS. Deploy the full stack locally using Docker Compose + LocalStack (simulated AWS). Deliberately introduce realistic code quality issues — missing documentation, anti-patterns, and placeholder credentials — to give the audit engine meaningful findings to discover.

Running Spring Boot service locally LocalStack: DynamoDB + SQS + S3 + EventBridge Solace broker in Docker Kubernetes manifests (EKS-compatible) Dockerfile + Helm chart Intentional audit targets seeded
PHASE 2 Week 2–3

Sanitization & Local Analysis Pipeline

Build the core security layer: the Java AST skeleton extractor and secret scanner. This module runs entirely offline — it reads Java source files, strips all method bodies and credential values, detects hardcoded secrets and anti-patterns locally (DynamoDB hot partitions, missing retry logic on MQ/Solace consumers, no dead letter queue configuration), and produces a structured skeleton safe for external transmission. No network calls in this phase.

Java skeleton extractor (AST-based) Secret scanner with airline-agnostic patterns DynamoDB anti-pattern detector (local) MQ/Solace consumer pattern linter Test coverage gap detector Pre-send validation gate
PHASE 3 Week 3–4

Claude Integration & Report Generation

Connect the sanitized skeleton output to the Claude API. Implement three analysis passes: architecture mapping (what is this system?), security and quality review (what needs fixing?), and final synthesis (write the three documentation outputs). All Claude responses return structured JSON — not prose — which is then rendered into final reports. Reports are saved locally and uploaded to the local S3 bucket via LocalStack.

Architecture analysis prompt + parser Security audit prompt + findings schema MQ / Solace topology analysis ARCHITECTURE.md generation SECURITY_AUDIT.md generation DEVELOPER_ONBOARDING.md generation S3 report upload (LocalStack)
PHASE 4 Week 4–5

Real AWS Promotion + Enterprise Migration Map

Deploy the identical pipeline to real AWS free tier. Demonstrate that zero application code changes are required — the environment variable switch from LocalStack to real AWS is the only change. Run the audit against the same microservice. Validate reports are identical. Produce the Enterprise Migration Architecture document showing how this scales to CodeBuild, EventBridge scheduling, Confluence publication, and JIRA ticket creation across all microservices in the estate.

Pipeline running on real AWS free tier Zero code change validation End-to-end demo recording Enterprise architecture document Cost projection for full rollout POC findings presentation

08 — Validation

POC Success Criteria

The POC is declared successful when all six criteria are met. These are binary pass/fail — not subjective assessments.

01

Secret Detection Accuracy

The sanitizer detects 100% of deliberately planted credentials (hardcoded passwords, API keys, connection strings) and blocks transmission if any are found. Zero false negatives accepted.

02

Architecture Document Quality

The generated ARCHITECTURE.md is reviewed by two senior engineers who were not involved in building the POC service. Both must rate it as accurate and useful for onboarding.

03

Finding Recall Rate

Claude's security and quality findings must surface at least 80% of the deliberately introduced anti-patterns (DynamoDB hot key, missing MQ retry, swallowed exceptions, etc.).

04

End-to-End Execution Time

Full pipeline — from CLI invocation to three completed reports in S3 — completes in under 8 minutes for a microservice of typical size (300–600 Java files).

05

Zero Code Change Promotion

The exact same pipeline code and Docker image runs against real AWS (free tier) by changing only environment variables. No code modifications required for the promotion.

06

Messaging Topology Detection

The system correctly identifies and documents the MQ/Solace topic and queue structure, consumer groups, and message flow between services from code analysis alone.

09 — Risk Assessment

Risks & Mitigations

Three risk categories identified. All are mitigated by design decisions already made in the architecture.

LOW RISK

Sensitive Code Exposure

Risk that proprietary source code or credentials are inadvertently transmitted to external AI service.

Mitigation: Sanitization layer is a hard architectural gate. Pipeline is blocked if pre-send validator finds any sensitive content. All secrets replaced with typed placeholders before any network call.
LOW RISK

AWS Cost Overrun

Risk of unexpected AWS charges during POC execution.

Mitigation: Phase 1–3 uses zero AWS spend (LocalStack local). Phase 4 uses AWS free tier only. Budget alert set at $5/month. All services used are within free tier limits for POC-scale usage.
MEDIUM RISK

AI Output Accuracy

Risk that Claude's architecture analysis contains inaccuracies that mislead engineers if treated as ground truth.

Mitigation: Success criteria require senior engineer review of outputs. Reports are explicitly labeled "AI-Generated — Requires Engineering Review." POC is a starting point, not a replacement for human judgment.
MEDIUM RISK

Solace MQ Topology Complexity

Risk that complex Solace topic hierarchies and MQ routing rules cannot be fully inferred from static code analysis alone.

Mitigation: Supplement static analysis with runtime topology export from Solace SEMP API. Treat Solace section of reports as "partial — requires runtime validation" in POC phase.
